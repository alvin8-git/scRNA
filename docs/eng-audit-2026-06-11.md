# Engineering Audit — Cache State Machine, Boundary Conditions, Execution Fallbacks

**Date:** 2026-06-11 · **Reviewer:** staff-eng deep-dive (Fable 5) · **Mode:** read-only audit, no code changes yet
**Scope:** the 14-step pipeline (01–10 core + 06b; 11–14 extracted to `pipeline/projects/bat_wing/`) — three axes requested after the structural product review.

Two framing corrections up front:
- This is **not a 10-step** pipeline anymore — 01–10 core plus 06b, with 11–14 under `projects/bat_wing/`.
- The `commandArgs()` fallback is **not in step 06**. Step 06 (and every core step 01–10) hardcodes `source("/data/alvin/scRNA/pipeline/config.R")`. The real `commandArgs()` resolver lives only in `validate_config.R` and the four `bat_wing/` scripts. This matters — see Axis 3.

## Headline finding (P1, latent regression from commit `52e5674`)

**`04_integrate.R` double-prefixes barcodes.** Step 01 already self-documents cell names:

```
01_load_qc.R  load_sample():  seu <- RenameCells(seu, add.cell.id = sample_name)   # "H1_AAACCTG..."
04_integrate.R merge():       add.cell.ids = SAMPLE_NAMES                          # prepends "H1_" AGAIN
                                                                                   # => "H1_H1_AAACCTG..."
```

The `add.cell.ids = SAMPLE_NAMES` change (the "self-documenting barcodes" P3 item) was based on a wrong premise: I assumed merge would generate generic `_1/_2` suffixes on barcode collision. It won't — step 01's `RenameCells` already prefixes every barcode with the sample name, so the cached objects entering step 04 are *already* collision-free and self-documenting. Re-prefixing produces `H1_H1_...`.

- **Why the recent run didn't catch it:** the 05→06→07 run reused an `integrated_seurat.rds` built by an *older* step-04 run (`add.cell.ids = NULL`), so barcodes were still single-prefixed. The bug is **latent** — it fires the next time step 04 runs.
- **Impact:** ugly double-prefixed barcodes; any external join keyed on barcode (or a human reading them) breaks. Internally consistent within a run, so it won't crash — it silently corrupts the barcode namespace.
- **Resolution:** revert `04_integrate.R:94` to `add.cell.ids = NULL`. The "cosmetic" P3 item was not just cosmetic, it was wrong; provenance was already handled by step 01.

## Axis 1 — State machine & cache validation

**Verdict: the cache correctly invalidates on threshold/array changes, and there is no halt/loop risk in the cache logic itself. Two real issues, both at the `validate_config.R` boundary.**

### Cache state machine (per-sample, steps 01–03 only)

```
 for nm in SAMPLE_NAMES:
   cache_path exists? ──no──► COMPUTE ─► save rds + write .hash(cache_hash(nm,step))
        │yes
        ▼
   stored .hash == cache_hash(nm, step)?
        │                         │
       yes                        no
        ▼                         ▼
   CACHE HIT (file.copy)     CACHE STALE ─► COMPUTE ─► overwrite rds + .hash
```

`cache_hash(nm, step) = md5( .cache_params(step) + resolved_input_path + .matrix_fingerprint(path) )`

- **Threshold vectors / array params DO invalidate.** `.cache_params` returns nested lists (`qc=QC`, `cluster=CLUSTER`, …); `digest::digest` hashes the whole structure, so changing `QC$max_features` or `CLUSTER$resolutions = c(...)` changes the md5 → STALE → recompute. Verified by per-step isolation test earlier (CLUSTER change moves only the step-03 key; QC change cascades to all three).
- **No infinite-loop risk.** The cache path is a linear `if/else` per sample. There is no loop that re-enters on staleness; STALE just recomputes once and rewrites the hash. The user's "infinite loop" concern is unfounded.
- **Coverage is correct.** Only steps 01–03 are cached, and `.cache_params` covers exactly what they depend on (QC → 01; +DOUBLET → 02; +NORM/DIM/CLUSTER → 03). Params that only affect 04+ (`HARMONY`, `MARKERS`, `SINGLER_REF`, `SUBCLUSTER`, `CLUSTER_CELLTYPE_MAP`) are deliberately absent because 04+ are not cached. No cached step depends on an un-hashed param.

### A1-1 (P2, known) — `validate_config.R` Check 3 halts on an unmounted drive
`validate_config.R:33-36` calls `dir.exists()` on every `SAMPLE_PATHS` entry and `quit(status=1)` if any is missing. If data lives on a NAS/external drive not yet mounted, validation halts before any config-only checks run. This is the pending `--skip-paths` / warning-mode item. **Recommendation:** demote the missing-path case to a warning (or gate it behind `--skip-paths`), keep checks 1–2 as hard errors.

### A1-2 (P3) — an embedded source-grep regression test lives inside the validator
`validate_config.R:38-41` `readLines("10_rarefaction.R")` and greps for a hardcoded `ES03_newkit` string. Two problems: (a) it couples the config validator to a specific past bug — that belongs in a test, not the pre-flight validator; (b) `readLines` throws an ugly raw error (not a clean `ERROR:` line) if `.this_dir` resolution misses or the file moves. **Recommendation:** move this to a real test (or a `tests/` smoke check); keep `validate_config.R` to config invariants only.

### A1-3 (P3) — fingerprint mtime resolution is whole-second
`.matrix_fingerprint` uses `as.integer(info$mtime)` (1-second resolution) + `info$size`. A sub-second rewrite that keeps identical byte size would not change the fingerprint → stale hit. Astronomically rare for 10x matrices; note it, don't fix unless it bites.

## Axis 2 — Boundary conditions & silent mutations

**Verdict: the explicit guards added in the data-integrity pass are solid (SingleR `anyNA` stop, `nrow(.scale_mat)==0`, `length(cells)==0 → NULL`, Read10X modality stop). The headline barcode bug is above. Two residual silent-mutation paths remain.**

### A2-1 (P3) — `which.max` on all-zero marker scores silently assigns the first cell type
`05_annotate.R:187` `data.frame(cluster=cl, sctype=names(which.max(cl_scores)))`. `cl_scores = colSums(.score_mat[cells,])`. The `length(cells)==0` guard (line 185) covers empty clusters, but if a cluster has cells yet **zero marker signal** (all scType scores 0 — e.g. a contaminant or a type with no markers in the panel), `which.max` returns index 1 → the cluster is silently labelled the **first** cell type in the panel. Same pattern at `:419` (subtype refinement) on all-equal scores. **Recommendation:** if `max(cl_scores) <= 0` (or all-equal), assign `"Unknown"` rather than an arbitrary first label. This is exactly the "silent corrupted label propagating downstream" the audit was asked to find.

### A2-2 (P3) — a cluster scType can't score silently vanishes from the consensus
`05_annotate.R:185` returns `NULL` for a cluster with no cells, so `.cl_sctype` (via `do.call(rbind, ...)`) drops that row. The consensus table is `merge(cluster_singler, .cl_sctype)` with default `all=FALSE` (inner join), so a cluster missing from `.cl_sctype` **disappears from `consensus_annotation.csv` and the pre-filled map entirely** — the user is never told it needs review. Also `merged$sctype_label <- .cl_sctype$sctype[match(...)]` (`:193`) yields `NA` for those cells (cosmetic on the sctype UMAP). **Recommendation:** left-join the consensus (`all.x=TRUE`) and emit dropped/`NA`-scType clusters as `REVIEW` with a `no-scType-score` note, so nothing silently disappears.

### A2-3 (P3) — `match(nm, cell_fate$Sample)` no-ops on a miss
`02_doublets.R:132` `idx <- match(nm, cell_fate$Sample)`; a miss returns `NA`, and R drops `x[NA] <- v` silently → that sample's doublet counts never land in `cell_fate`, with no error. Low impact (bookkeeping table), but it's a silent no-op. **Recommendation:** `stopifnot(!is.na(idx))` or build `cell_fate` keyed so the join can't miss.

## Axis 3 — Script execution & config-resolution fallbacks

**Verdict: there are THREE different patterns for "find config.R / pipeline dir," and the 10 core steps use the *least* robust one. This is the real fragility, and it's where the user's mental model is wrong.**

```
                          how it locates config.R / pipeline dir
  validate_config.R   ──► sys.frame ofile  ►  commandArgs --file=  ►  "."   (ROBUST, 3-tier)
  bat_wing/11..14     ──► commandArgs --file=  ►  "."                        (robust for Rscript)
  config.R (pdf_help) ──► sys.frame ofile  ►  "."                           (works when sourced)
  01..10 core steps   ──► source("/data/alvin/scRNA/pipeline/config.R")     (HARDCODED ABSOLUTE)  ◄── fragile
```

### A3-1 (P1/P2) — every core step hardcodes the absolute config path; `SCRNA_BASE_DIR` is half-implemented
`grep` confirms `01..10` all do `source("/data/alvin/scRNA/pipeline/config.R")`. Consequences:
- The `SCRNA_BASE_DIR` override (added so the pipeline can run against a NAS/relocated tree) is **only honored for data paths inside config.R**. The step scripts can't find config.R anywhere but `/data/alvin/scRNA/pipeline/` — so a relocated checkout breaks immediately, and `SCRNA_BASE_DIR` is a half-feature.
- Standalone `Rscript pipeline/05_annotate.R` works *only* if the repo physically lives at `/data/alvin/scRNA`.
- **This is the correction to the brief:** step 06 has no `commandArgs()` fallback — it hardcodes like the rest. The robust resolver exists in `validate_config.R`; the core steps never got it.
- **Recommendation:** extract the 3-tier resolver into one helper and have every step use it, OR (smaller diff) have each step compute its own dir from `commandArgs --file=` and `source(file.path(dir, "config.R"))`. Either way, kill the hardcoded absolute path. See DRY note below.

### A3-2 (P2) — standalone execution silently defaults to H1/H2 + human if env vars are unset
`config.R` falls back to hardcoded `SAMPLE_PATHS = {H1,H2}` when no `SCRNA_SAMPLE{N}` env vars are set (`:72-79`), and `SCRNA_SPECIES` defaults to `"human"` (`:400`). So `Rscript pipeline/05_annotate.R` with no env exports **silently** reads/writes `results_H1-H2_filtered/` with human references — no error, no warning — even if you meant a bat cohort. The wrapper sets these; standalone forgets them. **Recommendation:** when no `SCRNA_SAMPLE*` env var is present, emit a loud `message()` ("no SCRNA_SAMPLE env set — using hardcoded H1/H2 defaults") so a forgotten export is visible, not silent.

### A3-3 (P3) — bat_wing `commandArgs` resolver fails under `source()` (not `Rscript`)
`bat_wing/*.R` resolve `.pipeline_dir` from `commandArgs --file=`; if absent (interactive `source()`), it degrades to `"."` → `dirname(dirname("."))` = `".."` → `source("../config.R")` relative to cwd, which won't find config. Only matters for interactive sourcing (unsupported path); `Rscript` always sets `--file=`. Note, low priority.

## Cross-cutting

### X-1 (DRY, P2) — three resolvers for one problem
`validate_config.R`, the `bat_wing` scripts, `config.R`'s pdf_helpers source, and the (hardcoded) core steps each solve "find a sibling file" differently. That's a DRY violation and the root cause of A3-1. **Recommendation:** one `.resolve_pipeline_dir()` (the 3-tier `sys.frame ofile → commandArgs --file= → "."` logic), defined once and reused. Make the change easy (consolidate the resolver), then make the easy change (point every step at it).
